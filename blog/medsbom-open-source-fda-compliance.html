MedSBOM: Building an Open-Source FDA Compliance Tool

SBOM scanners are good at listing components and vulnerabilities. Medical-device teams still have to turn that output into reviewable evidence, risk decisions, and documentation that fits their quality process. I built MedSBOM as an open-source layer for that translation work.

One command should produce a traceable draft, not another spreadsheet to rebuild by hand.

The gap

Tools such as Syft, Grype, and Trivy can generate SBOMs or identify CVEs. Their output is deliberately general-purpose. A regulated product team still needs to record component context, vulnerability disposition, end-of-life status, review history, and the rationale behind risk decisions.

MedSBOM does not replace a quality system, regulatory review, or cybersecurity expertise. It creates structured drafts and an audit trail that teams can review inside those processes.

What the tool does

# Analyze an SBOM for vulnerability and lifecycle context
medsbom check firmware.sbom.json \
  --device "Example Device" \
  --device-version "2.3.1"

# Generate supported documentation formats
medsbom report firmware.sbom.json \
  --format all \
  --output ./review-package/

The workflow can produce draft artifacts for:

  • Premarket cybersecurity summary preparation
  • IEC 62304 SOUP-oriented component risk review
  • Timestamped vulnerability-review records

The processing pipeline

Six explicit stages

1. Ingest: parse CycloneDX or SPDX JSON from an existing scanner.

2. Match: correlate components with vulnerability and exploitation data.

3. Lifecycle: add end-of-support context where sources provide it.

4. Score: apply a documented, inspectable risk-scoring method.

5. Render: generate reviewable document drafts from controlled templates.

6. Record: append review activity to a local audit history.

Engineering decisions that matter

DecisionReason
Apache 2.0 licensePermissive use with an explicit patent grant.
SQLite audit historyPortable local evidence without requiring hosted infrastructure.
Network-blocked testsUnit tests cannot silently depend on external APIs or rate limits.
Transparent scoringReviewers can inspect how risk inputs influence the result.
Disclaimer in generated outputThe boundary between automation and accountable review remains explicit.

A platform bug the tests exposed

One failure came from treating an arbitrary JSON string as a possible file path. A long string could return false from Path.exists() on one platform and raise OSError on another. Detecting JSON content before attempting file operations made the behavior explicit and portable.

The broader lesson was equally useful: if a test suite can reach the public internet, a rate limit or outage will eventually turn a unit test into an integration test. MedSBOM blocks network access by default and mocks each permitted boundary intentionally.

Trust boundaries

  • Generated documents are drafts for qualified human review.
  • Source-data accuracy determines output accuracy.
  • Vulnerability feeds can be incomplete, delayed, or revised.
  • Using the tool does not itself establish FDA, IEC 62304, or QMS compliance.
